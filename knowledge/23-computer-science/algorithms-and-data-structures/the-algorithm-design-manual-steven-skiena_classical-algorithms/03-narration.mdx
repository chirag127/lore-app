# Narration

## The Big Idea

Let me tell you about the algorithms book that does not look like an algorithms book.

Open it at random. On page 42, you will find a story about Egyptian pyramids. On page 97, a story about an airline ticket pricing system that saved a company millions of dollars. On page 156, a story about a legal case where a man's life depended on whether a particular permutation was possible.

There are no theorems in this book. Steven Skiena says so himself: "You will not find a single theorem anywhere in this book." Not one. No lemma, no corollary, no formal proof. And yet this book has taught more working programmers how to solve real algorithmic problems than any rigorous textbook ever written.

How? By focusing on the hard part. The hard part is not understanding how quicksort works — the hard part is knowing that your problem *is* a sorting problem in the first place.

## Why the Catalog Works

The second half of the book is a catalog of 75 problems. Here is typical entry format:

> **Problem:** Maximum Clique
> **Input:** A graph G = (V, E).
> **Output:** The largest subset of vertices such that every pair is connected by an edge.
> **Status:** NP-hard.
> **Discussion:** This is the canonical hard problem on graphs. Use the Carraghan-Pardalos algorithm for exact solutions on sparse graphs up to ~500 vertices. For larger graphs, use the greedy heuristic (add vertices in order of decreasing degree) or the tabu-search approaches described in [BH87].
> **Implementations:** Available in C at algorist.com; also in the DIMACS challenge code.

That is it. Two to four pages per problem. You do not learn how to implement max clique from scratch. You learn:
- What the problem is called
- Whether anyone has solved it before
- Where to find existing code
- What approach works for your input size

This is exactly what a working programmer needs. Not a proof of NP-hardness. A pointer to the solution.

## The War Stories

The war stories are what make the book memorable. They are not polished case studies from academic papers. They are messy, honest accounts of Skiena's own consulting work, complete with dead ends and wrong turns.

In "Give me a Ticket on an Airplane," Skiena describes working with an airline on a fare-pricing problem. The naive approach — compute all possible routes — was too slow. The clever approach — sort the cities by some criterion and use that to prune the search — worked. The story shows that the algorithm was the easy part. The hard part was realizing that the problem reduced to a search over a sorted space.

In "Skiena for the Defense," he describes testifying in a criminal case where a gambling operation's records needed to be reconstructed from fragmented data. The solution involved sorting, hashing, and careful handling of missing data. The algorithm was trivial. The context was anything but.

These stories stick in your head. Years after reading the book, you will remember the pyramid story when you reach for Big Oh analysis, and the airline story when you reach for sorting.

## The Elephant in the Room: The Code Quality

I need to be honest with you. The C code in this book is bad. Variable names like a, b, c, x, y are common. Functions with side effects are not documented. Edge cases are sometimes glossed over. Henrik Norman called it "totally unacceptable and negligent of the author to have such poor code quality throughout the book."

This is a legitimate criticism. The code works — Skiena tested it — but it does not teach good coding practice. If you are learning algorithms *and* learning to write clean code at the same time, the code in this book will teach you bad habits.

My advice: read the code to understand the algorithm, then re-implement it yourself with proper names and structure. The code is a sketch, not a blueprint.

## Reading Experience

The book is 730 pages. Part I (the tutorial) is about 350 of those. You should read Part I linearly, one chapter per sitting. The chapters are short — maybe 30 pages each — and the war stories make them go quickly. Part II you should browse, not read. Keep the book on your desk and when you encounter a problem at work, flip to the catalog.

The tone is friendly but assumes you are literate in programming. Skiena does not explain what a linked list is. He explains why you would choose a linked list over an array, and how that choice affects your algorithm's complexity.

## Final Judgement

The Algorithm Design Manual is not the best algorithms book. It is not the most rigorous, the most complete, or the most mathematically satisfying. But it is the most *useful* algorithms book for a working programmer. It will teach you how to *think* about algorithms — which problems to solve, how to model them, when to stop looking for an exact solution and start approximating.

Keep it on your desk. Use it as a reference. When the catalog solves a problem that would have taken you three days to research on your own — and it will — you will understand why this book has been in print for thirty years.
