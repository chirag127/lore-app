# Core Concepts

## The Python Data Model

The first chapter is foundational. Everything else in the book is
an application of the ideas introduced here.

Python's data model is the set of protocols — implemented as special
methods, also called "dunder" (double-underscore) methods — that
your objects can define to integrate with language constructs. When
you write `len(x)`, Python calls `x.__len__()`. When you write
`for item in x`, Python calls `x.__iter__()`. When you write
`x + y`, Python calls `x.__add__(y)`. When you write `x == y`,
Python calls `x.__eq__(y)`. When you write `with x as ctx`, Python
calls `x.__enter__()` and `x.__exit__()`.

The data model is what makes Python feel consistent. A well-designed
class that implements the right special methods will work with
`len()`, `iter()`, `sorted()`, `in`, `for`, `with`, `bool()`,
`repr()`, `==`, `<`, and so on, *without* those constructs knowing
anything about your class. The implementer pays a small cost
(implementing a handful of methods); every Python developer
benefits forever after.

The dunder methods the book emphasizes as the "must implement" set:

| Method | Trigger | Purpose |
|---|---|---|
| `__repr__` | `repr(x)`, debugging, REPL | Unambiguous developer-facing string |
| `__str__` | `str(x)`, `print(x)`, `format(x, '')` | Human-friendly display string |
| `__eq__` | `x == y`, `x != y` | Value equality |
| `__hash__` | `hash(x)`, dict keys, set members | Hashability; required if you override `__eq__` |
| `__len__` | `len(x)`, `bool(x)` (default) | "Size" — used by truthiness if `__bool__` absent |
| `__bool__` | `bool(x)`, `if x:` | Truthiness |
| `__iter__` | `for item in x`, `iter(x)` | Iterator protocol |
| `__getitem__` | `x[i]`, `x[i:j]`, iteration fallback | Indexing and slicing |
| `__contains__` | `item in x` | Membership; falls back to `__iter__` |
| `__call__` | `x(args)` | Make instances callable |
| `__enter__` / `__exit__` | `with x as ctx:` | Context manager protocol |
| `__add__`, `__mul__`, etc. | `x + y`, `x * y` | Operator overloading |

The book's advice on `__repr__` and `__str__` is worth quoting:
`__repr__` should be unambiguous (you should be able to copy-paste
it back into the REPL and get an equal object, ideally); `__str__`
should be readable. When in doubt, implement only `__repr__` —
Python uses it as a fallback for `__str__`.

## Part I — Data Structures

### Sequences: More Than Just Lists

Python's sequence protocol is one of the most-used abstractions in
the language. `list`, `tuple`, `str`, `bytes`, `bytearray`,
`memoryview`, `array.array`, `collections.deque`, and (via
`__getitem__`) most user-defined containers all support the same
operations: `len()`, `x[i]`, `for item in x`, `x[i:j]`,
`x in seq`, `x + y` (concatenation), `x * n` (repetition), and
`seq.index(x)`.

The 2nd edition adds a careful treatment of when to choose which
sequence type. The default answer is `list` — but a `deque` is
better for FIFO queues, `tuple` for fixed records, `array.array`
for compact numeric storage, and `str` for text.

List comprehensions are the book's preferred notation. They are
clearer than `map`/`filter` for almost all cases:

```python
# Preferred
squares = [x * x for x in range(10) if x % 2 == 0]

# Less clear
squares = list(map(lambda x: x * x, filter(lambda x: x % 2 == 0, range(10))))
```

Generator expressions replace list comprehensions when the
sequence is only iterated once and may be very large. The syntax
is identical except for the parentheses.

Tuples are not just immutable lists. They are excellent *records*,
and tuple unpacking is one of the most underused features in
Python:

```python
# Tuple unpacking in for loops
for name, shares, price in portfolio:
    total += shares * price

# Extended unpacking with *
first, *middle, last = (1, 2, 3, 4, 5)  # first=1, middle=[2,3,4], last=5
```

### Dictionaries and Sets: The Backbone

Dictionaries are Python's most-used data structure. They are hash
tables with O(1) average-case insert, lookup, and delete. The 2nd
edition treats dict comprehensions as the default way to build
them, and unpacking with `**` as the default way to merge them:

```python
# Dict comprehension
prices_usd = {item: usd_to_eur(price) for item, price in prices.items()}

# Merge with unpacking
config = {**defaults, **user_overrides, **env_overrides}
```

Three dict-related techniques deserve attention. **OrderedDict**
is no longer needed (regular dicts are insertion-ordered since
Python 3.7). **Defaultdict** fills missing keys with a factory
result, eliminating `if key not in d: d[key] = []` boilerplate.
**Counter** is a `dict` subclass for counting hashables — `most_common(n)`
is one of the most useful one-liners in the language.

Sets are dicts without values. They implement the same hash-table
internals and have the same O(1) average case. Use them for
membership testing when you don't need the values, for deduplication,
and for set algebra (`|`, `&`, `-`, `^`).

### Unicode vs. Bytes

This is the chapter many readers say permanently changed how they
write code. The mental model is simple:

- `str` — human text. Sequence of Unicode code points.
- `bytes` — binary data. Sequence of bytes (0-255).
- `bytearray` — mutable binary data.

Text comes in, gets encoded to bytes for storage or transmission,
and gets decoded back to text on the way out. Inside your program,
work in `str`. At the I/O boundary, encode/decode.

```python
# Reading text
with open('data.txt', encoding='utf-8') as f:
    text = f.read()

# Reading binary
with open('data.bin', 'rb') as f:
    raw = f.read()
```

The two most common bugs are (1) opening a text file in binary mode
or vice versa, and (2) assuming the default encoding is UTF-8
(it's usually ASCII on Windows, which breaks non-ASCII text).
Always pass `encoding='utf-8'` explicitly.

### Data Classes

The `@dataclass` decorator (new in Python 3.7, expanded since)
generates `__init__`, `__repr__`, and `__eq__` from class
annotations. It is the modern default for "a class that is just
some fields":

```python
from dataclasses import dataclass, field

@dataclass
class InventoryItem:
    name: str
    unit_price: float
    quantity_on_hand: int = 0
    tags: list[str] = field(default_factory=list)
```

The 2nd edition also covers `typing.NamedTuple` (immutable
records with type hints) and `typing.TypedDict` (dict shapes for
JSON). The rule of thumb: `dataclass` for mutable records,
`NamedTuple` for immutable ones, `TypedDict` for dicts with a
fixed shape.

## Part II — Functions as Objects

### First-Class Functions

In Python, functions are values. You can store them in lists, pass
them to other functions, return them from factories, and inspect
their attributes at runtime. This is the foundation of the
decorator pattern, the strategy pattern, dependency injection,
and most of Python's libraries.

```python
# Functions as values
handlers = {
    'text': handle_text,
    'json': handle_json,
    'xml':  handle_xml,
}
handler = handlers[content_type]
result = handler(payload)
```

Functions have introspection: `func.__name__`, `func.__doc__`,
`func.__defaults__`, `func.__kwdefaults__`, `func.__code__`,
`func.__annotations__`. The 2nd edition uses these to show how
parameter signatures can be inspected at runtime — the foundation
of tools like FastAPI, Pydantic, and Click.

### Decorators and Closures

A decorator is a function that takes a function and returns a
function. It is the canonical way to add cross-cutting behavior
( logging, timing, caching, retry, authentication) without
modifying the wrapped function itself.

```python
import functools
import time

def clock(func):
    @functools.wraps(func)
    def wrapper(*args, **kwargs):
        t0 = time.perf_counter()
        result = func(*args, **kwargs)
        elapsed = time.perf_counter() - t0
        print(f'{func.__name__}({args}, {kwargs}) took {elapsed:.6f}s')
        return result
    return wrapper

@clock
def slow(n):
    time.sleep(n)
    return n
```

Closures capture variables, not values. This causes the famous
late-binding bug in loops:

```python
# Bug — all functions return 4
funcs = [lambda: i for i in range(5)]
print([f() for f in funcs])  # [4, 4, 4, 4, 4]

# Fix — capture the current value as a default argument
funcs = [lambda i=i: i for i in range(5)]
print([f() for f in funcs])  # [0, 1, 2, 3, 4]
```

Parameterized decorators are a three-level function structure: a
factory that takes parameters and returns a decorator, which
takes the function and returns the wrapper. The 2nd edition also
introduces `nonlocal` for rebinding enclosing-scope variables
in modern closures.

### Type Hints in Function Declarations

Type hints are metadata for static analysis (mypy, pyright, IDE
tooling) and documentation. They have no runtime effect unless
you call a tool like `typing.get_type_hints()` or use a runtime
validator like Pydantic.

```python
from typing import Iterable

def normalize(numbers: Iterable[float]) -> list[float]:
    return [(x - min(numbers)) / (max(numbers) - min(numbers))
            for x in numbers]
```

The book recommends hints where they clarify intent, and skipping
them where the types are obvious from the names. Hints are not a
replacement for tests, and they do not make Python statically
typed. They are documentation that happens to be checkable.

## Part III — Object-Oriented Idioms

### References, Mutability, and Garbage Collection

Variables in Python are *references* to objects, not boxes that
hold values. This distinction explains almost every "weird" Python
bug. Two variables can refer to the same object; mutating it
through one reference is visible through the other.

The `==` operator calls `__eq__` (value equality); the `is`
operator checks identity (same object). Default `__eq__` falls
back to `is`. For mutable types, default equality is rarely what
you want; override it. For hashable types, override `__hash__`
in lockstep, or your objects will silently stop being usable as
dict keys.

The `copy` module distinguishes shallow copy (new outer container,
same inner objects) from deep copy (recursively new). When in
doubt, use `copy.deepcopy`. When in more doubt, make your objects
immutable.

### Interfaces: Protocols and ABCs

The 2nd edition devotes substantial attention to the two ways
Python expresses interfaces. **Abstract Base Classes** (ABCs) from
`collections.abc` and `abc` declare a nominal interface: you
inherit from them, and `isinstance(x, Iterable)` is True only
because `Iterable` says so. **Protocols** from `typing` declare
a *structural* interface: `isinstance(x, Sized)` is True if `x`
has a `__len__` method, regardless of inheritance.

```python
from typing import Protocol

class SupportsClose(Protocol):
    def close(self) -> None: ...

def close_all(items: Iterable[SupportsClose]) -> None:
    for item in items:
        item.close()
```

The book's guidance: use Protocols by default (they are duck
typing with a name); use ABCs when you need to share
implementation through abstract methods, or when you want to
mark a class as officially part of a hierarchy.

### Operator Overloading

The book's golden rules of operator overloading:

1.  Do not invent meanings. `+` should do something addable.
2.  Make operators work the way callers expect. If `a + b` works,
    `a += b` should too, and `b + a` should give the same result
    if the operation is commutative.
3.  Make the implementer's life easy. The most useful operators
    (`+`, `*`, `==`, `<`) are easy to implement. The dangerous
    ones (`<`, `>`, `<=`, `>=`) are not — leave them alone unless
    you have a real use case.

The 2nd edition's chapter on operator overloading includes a
custom `Vector2d` class that implements arithmetic, comparisons,
`abs()`, `bool()`, hashing, formatting, and unpacking — a tour
of the dunder methods that makes a class feel native.

### Inheritance: Use It Sparingly

Inheritance is for "is-a" relationships where the subclass needs
the parent's implementation. It is not for code reuse; that is
what composition and mixins are for.

The book's recommended pattern: small mixin classes that each
contribute a single, well-named piece of behavior. `LoggedMixin`,
`PickleMixin`, `HashedMixin`. A class composes them by
inheritance, but no single base class grows into a god object.

## Part IV — Control Flow

### Iterators and Generators

Iteration is the most pervasive control-flow abstraction in
Python. The iterator protocol is two methods: `__iter__` returns
an iterator, `__next__` returns the next item or raises
`StopIteration`. `for` loops, list comprehensions, generator
expressions, `sum()`, `any()`, `all()`, `min()`, `max()`, `map()`,
`filter()`, `zip()` — all of them speak the iterator protocol.

Generators are a concise way to write iterators. A function with
`yield` is a generator factory. Each call returns a generator
object; each `next()` call runs the function until the next
`yield`. This makes lazy sequences trivial:

```python
def fibonacci():
    a, b = 0, 1
    while True:
        yield a
        a, b = b, a + b
```

Generators save memory (they don't build a list), support
streaming (infinite sequences are fine), and can be composed
with `for` and `itertools`.

### Context Managers

A context manager is an object with `__enter__` and `__exit__`.
The `with` statement guarantees `__exit__` runs, even on
exceptions. The classic use case is resource management: open
files, acquire locks, start transactions, manage database
connections, suppress expected exceptions, temporarily change
global state.

`@contextlib.contextmanager` lets you write a context manager
as a generator with a single `yield`:

```python
from contextlib import contextmanager
import time

@contextmanager
def timer(label):
    t0 = time.perf_counter()
    try:
        yield
    finally:
        print(f'{label}: {time.perf_counter() - t0:.3f}s')
```

### Coroutines and async/await

`async def` defines a coroutine function. Calling it returns a
coroutine object. `await` yields control to the event loop,
which runs other coroutines while the awaited task is waiting
on I/O. The 2nd edition walks through the evolution from
generator-based coroutines (`@asyncio.coroutine`, `yield from`)
to native coroutines (`async def`, `await`) and explains why
the change was made.

The book's guidance on `asyncio` is sober: it is a tool for
I/O-bound concurrency with many simultaneous connections. It
is *not* a way to make a single CPU-bound operation faster.
For that, use `multiprocessing`. For fire-and-forget I/O on
existing libraries, a `concurrent.futures.ThreadPoolExecutor`
is often simpler.

### concurrent.futures

`concurrent.futures` is a higher-level interface for thread and
process pools. `Executor.submit()` returns a `Future`; you can
attach callbacks, wait for completion, or iterate results as
they finish. The book recommends it for any "do N things
concurrently" problem where you don't need fine-grained
control of the event loop.

## Part V — Metaprogramming

### Properties

`@property` turns a method into a computed attribute. It is the
right answer for "a value that is computed from other attributes,
but I want callers to access it as if it were a field":

```python
class Circle:
    def __init__(self, radius):
        self.radius = radius

    @property
    def area(self):
        import math
        return math.pi * self.radius ** 2
```

The book's advice: properties are for computed values and for
adding validation to a public attribute. They are *not* for
hiding expensive computation — that is a job for caching
(`functools.cached_property`).

### Descriptors

Descriptors are the mechanism behind `property`, `classmethod`,
`staticmethod`, and method binding. A descriptor is a class
that defines `__get__`, `__set__`, or `__delete__`. When
Python evaluates `obj.attr`, it checks whether `attr` on the
class is a descriptor; if so, it calls the descriptor's
`__get__` instead of returning the class attribute directly.

The book's descriptor chapter is famously the densest in the
book. It rewards careful study. Once you understand descriptors,
you understand how `property`, `classmethod`, `staticmethod`,
and method binding all work, and you can build your own
descriptor-based APIs (the foundation of ORMs like SQLAlchemy
and Django models).

### Class Decorators and Class Metaprogramming

The 2nd edition devotes significant space to class decorators
and the new class-creation hooks (`__init_subclass__`,
`__set_name__`) introduced in PEP 487. These are the modern
preferred way to customize class creation. They replace
metaclasses for most practical use cases.

The book's rule of thumb: use class decorators first, reach
for `__init_subclass__` for subclass customization, and only
use metaclasses when you are building a framework that needs
to enforce a class-creation policy across many user classes.
