## Pythonic Thinking: The Habits That Matter Most

The first section is where the book does its best work. The opening
tricks establish a set of habits that are not syntax — they are the way
experienced Python developers *think* about common situations.

### Truthiness

Python's truthiness model is the first habit to internalize. In Python,
`[]`, `{}`, `""`, `0`, `None`, and `False` are all falsy. Everything
else is truthy. This is not a bug; it is a design choice, and it is
consistent.

The developer who writes:

```python
if user is None or user == {} or user == [] or user == "":
    return None
```

is fighting the language. The developer who writes:

```python
if not user:
    return None
```

is using it. Bader's discussion of this pattern — when to check for
`None` explicitly (because zero or empty is a valid value) and when to
use truthiness (because zero or empty is not) — is worth the price of
the book by itself.

### Chained Comparisons

Python's chained comparison is one of the most elegant features in the
language and one of the most underused. `0 < x < 10 < y` is not two
comparisons evaluated left-to-right with `x` and `y` checked twice. It
is parsed as a single predicate: `0 < x and x < 10 and 10 < y`, with
each variable evaluated at most once.

This enables bounds checks that read like mathematics and short-circuit
if any link fails. It is the correct form for any multi-bound check,
and Bader's explanation is the clearest treatment available.

### The `for`-`else` Construct

The `else` clause on a `for` loop is one of Python's most misunderstood
features. It does not mean "else in the loop" — it means "the loop
completed without a `break`." This makes it the canonical "search and
report if not found" pattern:

```python
for candidate in candidates:
    if candidate.is_qualified():
        selected = candidate
        break
else:
    raise ValueError("No qualified candidates found")
```

The `else` clause runs only if the loop did not `break`. It is
analogous to `try`/`else` — the `else` runs only if the preceding
block completed its contract. Bader's explanation of this pattern is
the clearest print treatment available.

## Effective Functions: Generators, Decorators, Context Managers

The second section covers the three patterns that most developers do not
fully internalize until they have been writing Python for years.

### Context Managers

A context manager is a block of code with guaranteed cleanup. Use it
whenever you acquire a resource — a file, a lock, a database
connection, a temporary directory — and need to release it regardless
of whether the block succeeds or raises.

The class-based form requires implementing two methods. The
`@contextlib.contextmanager` form requires writing a single generator:

```python
import contextlib

@contextlib.contextmanager
def managed_resource():
    resource = acquire()
    try:
        yield resource
    finally:
        release(resource)
```

The mental model: everything before `yield` is setup; everything after
`yield` is teardown. The `with` body runs *at* the `yield`. This is
the single most copied pattern in the book, and the most useful trick
most developers never learned in their beginner tutorial.

### Decorators

A decorator is a higher-order function that wraps a function. The `@`
syntax is sugar:

```python
@my_decorator
def my_function():
    ...
```

is exactly the same as:

```python
def my_function():
    ...
my_function = my_decorator(my_function)
```

The practical rules:
1. Use `@functools.wraps(fn)` on the inner wrapper to preserve
   `__name__`, `__doc__`, and `__module__` — without it, every
   decorated function shows up as `wrapper` in stack traces.
2. Use `*args, **kwargs` on the wrapper so the decorator is generic.
3. A parameterized decorator is three levels of nesting: factory →
   decorator → wrapper.

### Generators

A function containing `yield` returns a generator — a lazy iterator
that produces values on demand. This means:

- It never holds the full sequence in memory.
- It produces values as the caller requests them.
- It is the foundation of Python's `for`-loop, `itertools`, `sum()`,
  `max()`, and `asyncio`.

```python
def read_lines(path):
    with open(path) as f:
        for line in f:
            yield line.strip()

# No full file read into memory. One line at a time.
for line in read_lines("large_file.txt"):
    process(line)
```

## Classes and OOP: Data Classes and Dunder Methods

The third section covers the patterns that make classes feel native in
Python. The key insight: Python's object model is a set of protocols —
special methods (dunder methods) that classes implement to integrate
with the language's built-in behaviors.

### `__repr__` and `__str__`

`__repr__` is the canonical string representation of the object. The
unstated rule: "if it could be eval'd back to the same object, it is a
good `__repr__`." `__str__` is the human-readable form. Always
implement `__repr__`. Implement `__str__` only when it differs.

### Data Classes

`@dataclass` (PEP 557, Python 3.7+) generates `__init__`, `__repr__`,
`__eq__`, and `__hash__` from class field annotations. The boilerplate
removal is real:

```python
from dataclasses import dataclass, field
from typing import List

@dataclass
class Author:
    name: str
    books: List[str] = field(default_factory=list)
    active: bool = True

author = Author(name="Dan Bader", books=["Python Tricks"])
# Author(name='Dan Bader', books=['Python Tricks'], active=True)
```

The critical gotcha: `field(default_factory=list)` is the only safe
way to set a mutable default. `books: List[str] = []` shares the same
list across all instances — the classic Python mutable-default trap.

## Dictionaries, Sets, and Comprehensions

The section on dictionaries is extensive and practical. The most
important pattern: use `defaultdict` or `setdefault` to avoid the
`if key not in d` / `d[k] = ...` dance that appears in every beginner's
grouping code.

### Comprehension Syntax

```python
# List
squares = [x**2 for x in range(10)]
evens = [x for x in range(20) if x % 2 == 0]

# Set
unique_lengths = {len(s) for s in strings}

# Dict
title_by_author = {b.author: b.title for b in books}

# If/else inside a comprehension (conditional expression):
labeled = [f"{n}: even" if n % 2 == 0 else f"{n}: odd" for n in range(5)]
```

The rule: if the comprehension requires more than one `for` clause or
more than one `if` clause, or if it contains side effects, use a
`for`-loop. Comprehensions are for direct transformation. They are not
a replacement for all `for`-loops.

## Productivity Tools: f-strings, Type Hints, CLI, Packaging

The productivity section is the book's most distinctive contribution.
Almost no Python language book devotes a chapter to the tools Python
developers use daily — linters, formatters, type checkers, CLI
frameworks, and packaging.

### f-strings

Python 3.6 added f-strings (PEP 498). They are the fastest, most
readable string formatting in the language:

```python
name = "Dan Bader"
count = 47

msg = f'{name} has {count} Python tricks'
# f'{name.upper()} has {count ** 2} tricks: {name!r}'
```

f-strings support arbitrary expressions, conversion flags (`!r`, `!s`,
`!a`), and format specifiers. They are faster than `%` formatting and
`str.format()` because they are evaluated at compile time where
possible.

### The Walrus Operator

PEP 572 added the assignment expression (`:=`). It assigns inside an
expression, reducing repeated function calls:

```python
# Without walrus:
if len(items) > 10:
    print(f"Processing {len(items)} items")

# With walrus:
if (n := len(items)) > 10:
    print(f"Processing {n} items")
```

The crucial grammar rule: the assignment expression *must* be
parenthesized when not at the top level of an expression. This is a
parser constraint, not an optional style choice.

### CLI with Typer

The book's CLI section introduces `Click` (the decorator-based CLI
framework) and `Typer` (Click built on type hints, auto-generating
`--help`, `--version`, and shell completion):

```python
import typer

app = typer.Typer()

@app.command()
def hello(name: str, count: int = 1):
    """Say hello N times."""
    for _ in range(count):
        print(f"Hello, {name}!")

# `python cli.py hello --name Dan --count 3`
# `python cli.py hello --help`
```

Zero `argparse`. Zero `if __name__ == "__main__"` boilerplate. Type
hints generate the interface.

### Packaging with `pyproject.toml`

Modern Python packaging requires a `pyproject.toml` and a build
backend. `flit` is the simplest choice for a single-module package.
The flow: write a module, add `pyproject.toml`, run `flit build` and
`flit publish`:

```toml
[build-system]
requires = ["flit_core >= 3.2,<4"]
build-backend = "flit_core.buildapi"

[project]
name = "my-package"
version = "0.1.0"
description = "A short description"
readme = "README.md"
requires-python = ">=3.9"
license = {text = "MIT"}
authors = [{name = "Dan Bader", email = "dan@example.com"}]
dependencies = [
    "click >= 8.0",
]
```

Two commands. The package is on PyPI. The book emphasizes this
workflow because the old `setup.py` / `setup.cfg` / `requirements.txt`
triple is now obsolete.

## The Unified Model

The book's most important contribution is not any single trick — it is
the mental model that emerges when you read them all together. The
model is:

**Python is built on a small set of protocols that compose cleanly.**

The iterator protocol (`__iter__`, `__next__`) makes `for`-loops work
and underpins generators and `itertools`. Generators make context
managers (`yield` as a resource boundary) and `asyncio` (lazy
coroutines) trivial to write. Decorators wrap any function. Dunder
methods make your classes feel like native Python types. `@dataclass`
removes the boilerplate so you can focus on the fields.

Once you see these protocols as composable primitives — not as
separate features to memorize — the language becomes simpler. You stop
thinking "how do I write a context manager" and start thinking "I need
a `try`/`finally` block with a `yield` in the middle."

## Narrative Style and Delivery

**Narrator:** I keep two books on my desk for the moments when I am
not sure how to write something correctly. One is *Fluent Python*, and
I read it when I want to understand *why* something works. The other
is *Python Tricks*, and I reach for it when I want to know *how* to do
something — or when I am pairing with a developer and I want to show
them a pattern in 30 seconds instead of explaining it for 10 minutes.

The distinction matters. *Fluent Python* teaches you to think in
Python. *Python Tricks* teaches you to *write* Python. They are
complementary, not redundant.

**Narrator:** The thing most developers underestimate is how much time
they spend on the 15 patterns in this book. How many times have you
written a context manager without `@contextlib.contextmanager`? How
many decorated functions are missing `@functools.wraps`? How many times
have you looped over `enumerate(zip(...))` and wished you had a cleaner
syntax? These are not advanced Python patterns. They are the patterns
you use on Tuesday afternoon.

This is also why the book pairs so well with *Effective Python*. Slatkin
writes 90 items. Bader writes 15 to 20. Both are correct. Both are
useful. The difference is density: *Effective Python* is a catalog of
every idiom you might encounter. *Python Tricks* is a curated tour of
the ones you will use most, with more explanation and more narrative
per pattern.
