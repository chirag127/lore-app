## Introduction

Welcome to BookAtlas. Today: *A Philosophy of Software Design* by John
K. Ousterhout. Published 2021, second edition. Yaknyam Press. 196
pages. One of the shortest, densest, most opinionated books on
software design you will ever read.

This book has a thesis. One thesis. Every chapter serves it. Here it
is: the fundamental problem in software is complexity. Everything else
is secondary. And the way to fight complexity is to build deep
modules — modules where the interface is much simpler than the
implementation. That's it. That's the whole book.

Let's unpack it.

---

## Who Is John Ousterhout?

Before we talk about the book, we should talk about the author. John
Ousterhout is not a consultant who writes about code. He is a
distinguished engineer who *built* things. He created the Tcl scripting
language. He built the Tk toolkit. He led the team that designed the
Sprite operating system and the first log-structured file system. He
co-invented the Raft consensus algorithm. He founded two companies.

He has been a professor at Berkeley and Stanford, and spent 14 years
in industry. When he tells you how to design software, he is not
theorizing. He is reporting from decades of building production
systems.

This matters because the book's authority comes from Ousterhout's
credibility. You may disagree with him. But you cannot dismiss him as
someone who has never shipped.

---

## The Problem: Complexity

Ousterhout defines complexity as anything that makes code hard to
understand or modify. Two causes: dependencies and obscurity.

**Dependencies.** You cannot change A without understanding B. If your
system has 20 modules and each depends on the other 19, you cannot
hold the system in your head. You make changes by guesswork.

**Obscurity.** The code does not tell you what you need to know. Why
is this variable here? What is this method supposed to do? What
happens if I pass null? The information exists somewhere — in a
comments, in someone's memory, in a deleted Slack thread — but it is
not obvious from reading the code.

Complexity is not a single explosion. It is a thousand small
accumulations. A special case here. A pass-through variable there. A
vague name. Each one is harmless. Together, they make the system
unchangeable.

---

## The Solution: Deep Modules

The core idea. Visualize a module as a rectangle. The area is the
functionality. The top edge is the interface — everything someone must
know to use the module.

A deep module has a small top edge and a large area. A shallow module
has a large top edge and a small area.

The canonical deep module: Unix file I/O. Five functions. `open`,
`close`, `read`, `write`, `lseek`. That is the entire interface.
Behind it: hundreds of thousands of lines of code managing disk
sectors, block caching, device drivers, permissions, journaling. Most
developers never see any of it. They call `read`, they get bytes. The
complexity is hidden.

The canonical shallow module: the Java `FileInputStream` class. It
reads bytes. If you want buffering, you wrap it in a
`BufferedInputStream`. If you want data types, you wrap it in a
`DataInputStream`. Each wrapper is shallow individually. Together
they create a maze of tiny classes that the developer must navigate.

Ousterhout calls this **classitis**: the belief that classes are good
so more classes are better. Classitis produces systems where you spend
more time navigating interfaces than writing logic.

**Narrator:** I have been in codebases like this. You open a file and
find 47 classes, each 30 lines, each depending on three others. The
author thought they were being clean. They were being clean in the
wrong dimension. The classes were small; the system was unmanageable.

## Information Hiding

How do you make modules deep? Information hiding. The concept comes
from David Parnas, 1972. Each module should encapsulate a few design
decisions. Other modules should not know about them.

This reduces complexity in two ways. First, it simplifies the
interface — the caller does not need to know the internals. Second, it
enables independent evolution — you can change the hidden decision
without touching other modules.

The Unix filesystem has been completely rewritten three times over.
The interface — five functions — never changed. That is information
hiding at its best.

The opposite is **information leakage**: the same design decision is
spread across multiple modules. Both know about it. Both must change
if it changes. The knowledge is not hidden; it is duplicated.

**Narrator:** Every time you find yourself changing two files because
of a change in one concept, you have information leakage. The fix is
usually to merge the modules or to extract the shared knowledge into a
new module that hides it.

## Red Flags: What to Look For

Ousterhout gives concrete symptoms of poor design:

**Pass-through methods.** A method that does nothing but call another
method with the same signature. It adds complexity without adding
value. If you have a chain of these — controller calls service, which
calls repository — ask whether each layer is providing its own
abstraction or just forwarding.

**Pass-through variables.** A variable passed through multiple methods
that do not use it, just to reach the one that does. The variable
pollutes every interface in the chain. Store it in a context object or
restructure so the end consumer accesses it directly.

**Temporal decomposition.** Modules organized by execution order
instead of by knowledge. ReadInput -> ParseInput -> ProcessInput ->
WriteOutput. Each module knows too much about the overall flow.
Decompose by what the system knows, not by when it happens.

## Defining Errors Out of Existence

Ousterhout's most provocative practical advice: the best way to handle
exceptions is to make them impossible.

The example he returns to: Unix file deletion. In most operating
systems, if you try to delete a file that a process has open, you get
an error. In Unix, the file is unlinked from the directory but the
data remains accessible to the open handle. When the last handle is
closed, the storage is reclaimed. No error. The designer changed the
definition of deletion so the error could not occur.

Java's `substring` method does something similar. `"hello".substring(3,
3)` returns `""` instead of throwing. The designers could have made
this an error. They chose to make it a valid operation.

**Narrator:** This is a genuinely useful mindset shift. When you find
yourself writing an exception handler, stop and ask: can I change the
design so this situation never arises? Not always — sometimes errors
are real — but more often than you think, the answer is yes.

## Controversies: Where Ousterhout Parts Ways

Ousterhout is not a consensus builder. He picks fights.

**Against small methods.** Clean Code says methods should be 4-6
lines. Ousterhout says that leads to shallowness and fragmentation.
Depth matters more than length.

**Against TDD.** He argues TDD encourages tactical programming —
implementing one test case at a time without designing the
abstraction. He prefers designing the full module first.

**Against self-documenting code.** He argues the phrase is used to
justify not writing comments. Good comments are not optional. They
document the abstraction, which code cannot.

**Against declaring variables by interface type.** He says `ArrayList`
list = new ArrayList()` communicates intent better than `List` list =
new ArrayList()` because the caller often needs the specific
performance guarantees.

Each of these positions has defenders and detractors. The value of the
book is not that Ousterhout is right about all of them. The value is
that he forces you to think about why you hold your own opinions.

---

## The Verdict

**Narrator:** I have been a professional software engineer for 15
years. I have read most of the canon. This is the only design book
that changed how I write code. Not because it taught me something I
did not know — information hiding is half a century old — but because
it connected the dots. It gave me a single question to ask: is this
module deep?

That question is worth more than a dozen patterns. It applies at every
level — method, class, package, system. It is diagnostic and
prescriptive. It tells you what is wrong (shallow module) and what to
do about it (hide more, simplify the interface).

The book is not perfect. It is too short. The evidence is anecdotal.
Some of the controversial positions are overstated. And the advice
works best for systems software; application developers will need to
adapt.

But if you read one book on software design this year, make it this
one. Then go read Clean Code and decide for yourself who is right.

Design* by John K. Ousterhout. Thanks for listening.
