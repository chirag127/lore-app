## Strengths

- **Introduced patterns to software engineering.** Before this book,
  there was no systematic catalog of design solutions. The book
  created the field.
- **Timeless principles.** "Program to an interface, not an
  implementation" and "favor composition over inheritance" are as
  relevant today as in 1994.
- **Excellent pattern format.** The consistent structure (Intent,
  Motivation, Applicability, Structure, Participants, Consequences)
  became the industry standard.
- **The consequences section is the most important part.** Each
  pattern's trade-offs and liabilities are honestly discussed.

---

## Weaknesses

- **Dated examples.** The C++ and Smalltalk code examples feel archaic
  to modern readers.
- **No functional programming.** The book is purely OOP. Functional
  patterns (monads, lenses, etc.) are entirely absent.
- **Overly formal tone.** The writing is dry and academic. Many
  readers find it hard to get through.
- **Missing modern patterns.** The book predates dependency injection,
  MVC as UI pattern, repository pattern, and many others.

---

## Criticism

- **Patterns can be used to justify over-engineering.** The existence
  of a pattern is sometimes taken as justification for adding
  unnecessary complexity.
- **Some patterns are trivial in modern languages.** Iterator is
  built into every collection. Singleton is a module in Python and
  an object literal in JavaScript.
- **The book does not discuss when NOT to use patterns.** Beginners
  often apply patterns everywhere without considering whether they
  are needed.

---

## Comparison

| Book | Author | Focus |
|------|--------|-------|
| *Design Patterns* | Gamma et al. | The original 23 GoF patterns |
| *Head First Design Patterns* | Freeman et al. | Accessible Java-based introduction |
| *Patterns of Enterprise Application Architecture* | Fowler | Enterprise-level patterns |
| *Refactoring to Patterns* | Kerievsky | How to introduce patterns into existing code |
